etcd: 把 Raft 包装成 k8s 的事实数据源

Created2026-06-09|Updated2026-06-09
|Post Views:
📝 术语别名
  • etcd: 一个分布式、强一致、高可用的 KV 存储, 底层用 Raft 做共识, Go 写的, 现属 CNCF, 是 k8s 唯一的事实数据源 (source of truth);
  • revision: etcd 集群级、全局单调递增的版本号, 每次成功的写/事务 +1, 是整个 keyspace 的"全局时钟";
  • MVCC: Multi-Version Concurrency Control, 多版本并发控制, 写不覆盖旧值而是生成新版本, 让读写互不阻塞;
  • Watch: 订阅某个 key/区间从指定 revision 起的变更, 按序、不丢地推送给客户端;
  • Lease: 带 TTL 的 key, 客户端要不断续约 (keepalive), 断了自动删, 用来做心跳/选主;
  • Txn: etcd 的迷你事务, If/Then/Else, 本质是基于 revision 的 CAS (compare-and-swap);
  • compaction: 压缩, 丢弃某 revision 之前的历史版本以回收空间, 压缩点同时是 Watch 能回放的最早边界;
  • reconcile: 调谐, 比较"期望状态"与"实际状态"并采取动作消除差距的幂等操作;
  • List-Watch: 先全量拉一次基线 (List) 再持续追增量 (Watch) 的模式, k8s informer 的核心范式;

从 Raft 到 etcd: 中间多了什么

上回书我们学完了 Raft, 知道 Raft 解决的是 「一组节点如何就一条有序日志达成一致」 —— 它是个 replicated state machine 的框架, 但它不规定状态机到底是什么, 只保证所有副本以相同顺序看到相同的 log entry;

etcd 的回答是:

那条 Raft 日志里的每一条 entry, 就是一次 KV 写操作; 状态机就是一个 MVCC 的键值存储 (底层 bbolt);

所以分工很清楚: Raft 负责"所有副本以相同顺序看到相同的写", etcd 负责"写的是什么、能查到什么、上层暴露什么 API"; etcd 真正值钱的不是它内嵌了 Raft (那是基础), 而是它在 Raft 之上加了一组让分布式协调变得好写的能力 —— 这也是 k8s 选它而不是自己撸一套 Paxos 的根本原因;

能力 是什么 k8s 为什么离不开
MVCC + revision 全局单调递增版本号, 旧版本保留直到 compaction 是 Watch / 一致读 / CAS 的共同地基
Watch 从指定 revision 起订阅变更, 不丢不乱序 informer 的命根子, 驱动 reconcile
Lease 带 TTL 的 key, 断约自动删 节点心跳、组件 leader election
Txn 基于 revision 的 CAS 迷你事务 乐观并发控制

版本提醒: 现在都是 v3 API (gRPC + protobuf + 扁平 MVCC keyspace); v2 是老的 REST + 层级结构, 已弃用, 看资料别看串了;

MVCC: 给存储装上一条时间轴

它要解决的矛盾: 读写并发不互相阻塞

没有 MVCC 的老办法, 数据只有"一个当前值", 一个事务在写另一个想读同一份数据就冲突, 只能上锁 —— 读者等写者、写者等读者, 并发上不去;

MVCC 的办法: 每次写不是"改掉旧值", 而是"生成一个新版本", 旧版本保留; 每个读看到的是某个时间点的一致快照, 各读各的版本, 于是读者不阻塞写者、写者不阻塞读者 —— 因为它们碰的根本是不同版本; PostgreSQL / MySQL InnoDB / Oracle 全这么干;

etcd 里的 MVCC 长什么样

  1. 一个集群级、全局单调递增的 revision, 每次写/事务成功 +1;
  2. 底层 bbolt 存的不是 key → value, 而是 (key, revision) → value, 所以历史版本天然共存;
  3. 每个 key 还带三个数:
    • create_revision: 被创建时的 revision;
    • mod_revision: 最后一次修改的 revision ← CAS/Txn 就靠比它;
    • version: 被改过几次;
  4. 读可以 etcdctl get x --rev=6 直接读历史快照;

一个例子, 对 key x 的操作序列:

不同读者: “读最新” 看到已删除; “读 as of rev 6” 看到 x="b"; “读 as of rev 3” 看到 x="a"; 同一份数据同一时刻, 带着各自版本视角, 看到不同结果且都自洽;

代价是旧版本越堆越多, 所以要定期 compaction 丢弃老版本回收空间 —— 但压缩之后那些老快照就读不到了, Watch 也没法从更早的点续传 (后面会再撞上这一点);

一句话: MVCC 让 etcd 从"一个会用 Raft 同步的当前值存储", 升级成"一个带时间轴的存储" —— 而 k8s 需要的恰恰是这条时间轴: 谁在哪个 revision 改了什么;

Watch: 时间轴上的游标 + 重放引擎

在做什么

订阅一个 key 或一段区间 (前缀), 从某个 revision 开始, 把之后所有变更按序、不丢地推给你;

“从某个 revision 开始"正是 MVCC 的直接产物: 因为旧版本还在, 才能让你"回到过去某个点开始订阅”, 而不只是订阅未来;

API 形态 (v3, gRPC streaming)

Watch 是一条双向 gRPC 流, 不是一次性请求:

一条 gRPC 流可以复用多个逻辑 watcher (各自一个 watch_id); DELETE 事件的 kv.value 为空 (对应那个 tombstone);

三条核心保证 (一定要记)

保证 含义
Ordered 有序 事件严格按 revision 递增, 绝不乱序
Reliable 可靠 只要没被 compaction 干掉, watched 区间内的事件一个都不会被悄悄丢
Atomic 原子 同一 revision 产生的多个事件, 一定在同一个 WatchResponse 里整批交付

第三条很关键: 一次 Txn 可能同时改多个 key, 它们共享同一 mod_revision, Watch 保证打包给你, 你不会读到"半个事务";

内部实现: synced / unsynced 双队列

服务端用 watchableStore 包住 MVCC store, 把所有 watcher 分两组:

  • synced (已追平): start_revision 已赶上当前 revision; 走热路径, 每次写提交后在 notify 流程里直接推, 延迟最低;
  • unsynced (落后): start_revision 还停在过去 (比如 watch from rev 100 但现在已 rev 5000); 一个后台 goroutine syncWatchers 周期性从 bbolt 按 revision 区间扫历史事件、过滤出落在 watch 区间内的补发, 追平后把它从 unsynced 挪进 synced;
  • victim/慢消费者: 若接收 channel 满了 (你处理太慢), etcd 把该 watcher 丢进 victim 列表缓冲、之后重试, 避免一个慢 watcher 拖死整个写路径;
  • 底层支撑: 内存里有个 treeIndex (B-tree) 把 用户 key → keyIndex(该 key 历史所有 revision) 映射起来; 补历史时拿 revision 区间去 bbolt 扫, 再用 treeIndex 过滤;

和 compaction 的致命交互

如果 start_revision <= compact_revision (你想从的起点已被压缩), 服务端没法重放那段历史, 于是返回 ErrCompacted、把 watch 取消掉 (canceled=true);

客户端唯一出路: 重新拉一次全量当前状态, 拿到新 revision, 再从新 revision 重新 watch —— 记住这个模式, k8s 的 relist 就是它;

断线续传

客户端始终记住最后收到事件的 revision = R, 重连时用 start_revision = R+1 重建 watch (+1 因为 R 那条已收到过); progress_notify 对很久不变的 key 周期性推一个空 WatchResponse 只带当前 revision, 告诉你"到这点为止没漏", 让 R 能跟着推进, 避免回退太多触发 compaction;

Watch 的应用目的: 用事件流取代轮询

讲完机制, 退一步看为什么要有它; 一句话: 让分布式组件从"主动轮询共享状态"变成"被动接收变更", 把状态同步变成事件驱动;

没有 Watch 只能轮询, 而轮询有个死结:

  • 间隔短 → 实时但把 etcd 打爆 (惊群);
  • 间隔长 → 负载低但延迟高;
  • 更糟: 两次轮询之间改了又改回去, 你完全察觉不到中间发生过什么 (漏更新);

轮询的根本缺陷是只能看到"现在的快照", 看不到"发生过的事件序列"; Watch 用 “推送 + 有序不丢 + 可从历史续传” 一次性补齐, 于是应用能安全构建:

本地维护一份共享状态的实时镜像 + 每次变更触发一段反应逻辑 —— 即 List-Watch 模式;

落到具体应用:

应用 Watch 的目的
动态配置 watch 配置 key → 一改全实例热加载, 不重启不轮询
服务发现 watch /services/ 前缀 → 实时维护活的 endpoint 列表
Leader election watch leader key → leader 挂了立刻感知并接管
分布式锁/队列 watch “排我前面那个 key” → 它释放时精确唤醒我
k8s 控制器调谐 watch 期望状态 → 一变触发 reconcile, 驱动实际趋近期望

reconcile: 控制器调谐到底是什么

最后一行是 k8s 选 etcd 的根本理由, 单独展开;

reconcile 这个词的本义

它在会计里就是**“对账、使一致”**: 你 APP 里记的余额和银行对账单应该一样, 不一样就逐项核对调整直到吻合; k8s 把这个词原封搬来, 两份"账"换成:

  • 期望状态 (desired state): 你声明的、存在 etcd 里的 (“我要 3 个 Pod”);
  • 实际状态 (actual state): 集群里真在跑的 (“现在只有 2 个 Pod”);

reconcile 就是一个循环跑的函数

例子, Deployment 声明 replicas: 3:

控制器 = 一个不停跑 reconcile 的死循环 (control loop);

关键洞察: level-triggered 而非 edge-triggered

风格 含义 问题
edge-triggered 边沿触发 针对"发生了什么变化"反应 (“刚有人删了 Pod, 我补一个”) 漏一个事件, 状态就永久错
level-triggered 电平触发 针对"当前是什么状态"反应 (“不管发生过啥, 我现在看一眼差几个补几个”) 漏事件无所谓, 下轮重算就修正

k8s 的 reconcile 是 level-triggered 的; 这解释了为什么 Watch 偶尔漏个事件、或 relist 一下不致命:

Watch 事件在这里只是一个**"醒一醒, 去重新对一次账"的闹钟**; 真正干活的 reconcile 每次都重新读全量期望 + 重新观察实际 + 重算差距; 所以漏了事件, 下次任何事件 (或定时 resync) 触发 reconcile 照样修正;

这也要求 reconcile 必须幂等: 算出"差 0"时什么都不做, 重复跑无副作用;

和别处见过的 reconcile (Terraform plan/apply、React 协调虚拟 DOM、Git merge) 本质同一模式: 比较"想要的"和"现有的", 生成并执行抹平差异的动作; k8s 只是把它做成集群级、永续运行的基础范式;

谁在 watch 谁: Pod 需要自己维持 Watch 吗

一个常见误解 —— 直接结论:

普通应用 Pod 里的程序完全不需要、也根本不参与 Watch; Watch 是 k8s 控制面在"外面"自动维持的, 你的 Pod 对此一无所知;

两个关键事实:

  1. 只有 kube-apiserver 直连 etcd; 你的 Pod 永远不碰 etcd, 连地址都不知道;
  2. 干 Watch 活的是控制面组件 (controller-manager 里各种控制器、scheduler、kubelet), 它们 watch 的还是 apiserver (apiserver 再从 etcd 拿, 中间有 watch cache 挡着, 否则几千个 informer 会把 etcd 打穿);

心智模型纠正: 你的 Pod 是"账本里被对账的一行 (actual state)", 不是"对账的人"; Pod 崩了是控制器 watch 到"少了一个"然后补一个 —— 不是 Pod 自己 watch 自己救自己; 所以应用层无需写任何 watch、无需心跳上报、无需维持连接, 这正是声明式的爽点: 应用无感;

唯一例外: 你自己写控制器/Operator (ingress-controller、各种 Operator、service mesh 控制面) 时, 这个 Pod 本身就是个 watcher; 但即便如此 —— ① 它 watch 的仍是 apiserver, 绝不直连 etcd; ② 你不用手撸 gRPC watch 流/重连/relist, 用官方 client-go informer 框架: 注册几个事件回调 + 一个 reconcile 函数, watch 的生命周期 (连接、重连、relist、本地缓存) 全由库自动维持, 你只写"差距出现时干什么";

k8s 怎么把这套吃下去 (List-Watch 全景)

informer/reflector 几乎是 Watch 语义的教科书式用法:

  1. List: 先全量拉一次某资源 (如所有 Pod), 拿到 resourceVersion = RV0; resourceVersion 本质就是 etcd revision (对上层是不透明字符串);
  2. Watch: 从 RV0 + 1 开始 watch, 增量事件灌进本地缓存 (Indexer/Store), 控制器从本地缓存做 reconcile —— 这就是调谐数据的来路;
  3. Relist (兜底): watch 报 410 Gone / too old resourceVersion (底层就是 etcd ErrCompacted), 说明落后到被压缩了 —— reflector 丢弃缓存、重新 List、重新 Watch; 和前面 Watch 那节的续传模式一模一样;

两个 k8s 特有优化:

  • Watch Cache: apiserver 在内存维护最近事件的环形缓冲, 几千个 informer 从 cache 服务而非各自打穿 etcd;
  • Bookmark 事件: apiserver 周期性发一个只带 resourceVersion、没有实际变化的 bookmark, 让空闲 watcher 也能推进自己的 RV (同 etcd 的 progress_notify 思路), 减少 relist;

小结: 一条链路

etcd 的角色: 不是"存数据的数据库", 而是"整个集群协调逻辑的事件骨架"; MVCC 提供时间轴, Watch 把时间轴变成可靠事件流, reconcile 在事件流上做幂等对账 —— 三者叠起来, 才让 k8s “你声明期望、系统自己收敛” 这件事在工程上成立;
