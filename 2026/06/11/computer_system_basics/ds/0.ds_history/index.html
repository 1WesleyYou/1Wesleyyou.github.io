0. 分布式系统编年史: 2000-2020

Created2026-06-11|Updated2026-06-11
|Post Views:

这二十年其实是同一个故事在反复上演: 数据先撑爆了存储和计算, 然后撑爆了数据库, 接着业务拆成微服务又撑爆了运维; 每一次"撑爆"都逼出一批框架, 而每个框架背后都站着一家被现实摁在地上摩擦的公司; 这篇按时间把它们串起来, 每个框架只记三件事: 哪年、谁家、为解决啥需求; 它同时也是 ds/ 这一摞笔记的索引页, 相关的我都挂了内链;

一张总图先压个轴, 后面分代细讲:

一、2000-2006: 海量数据把 Google 逼出三驾马车

2000 年互联网泡沫一破, 一地鸡毛里活下来的是 Google、Amazon、Yahoo 这几家; Google 摊上的麻烦最典型: 它要把整个互联网抓下来建索引, 但买的全是会随时坏的廉价 PC; 于是它的核心需求就一句话: 怎么在一堆动不动就挂的破机器上, 可靠地存和算 PB 级数据;

这个需求直接催生了三篇奠基论文, 也就是大名鼎鼎的"三驾马车":

  • GFS (Google File System, SOSP 2003): 需求是存超大文件、且默认磁盘会坏; 做法是大文件切块、三副本、一个 master 管元数据; 它定下了"故障是常态不是意外"这个后来云原生的祖训;
  • MapReduce (OSDI 2004): 需求是让不懂分布式的普通工程师也能并行处理 PB 数据; 做法是把计算抽象成 map + reduce 两步, 容错/调度/数据搬运全藏在框架里; 这是"把分布式复杂性藏进框架"这个套路的鼻祖;
  • Bigtable (OSDI 2006): 需求是 Search/Maps/Gmail 要 web 尺度的结构化存储; 它是建在 GFS 上的稀疏大表;

而这三个都要选主、要知道"谁是 master"、要存元数据, 于是同年又出了 Chubby (OSDI 2006): 一个粗粒度锁服务, 专门给上面这些做协调; 详见 11.chubby; 它就是后来 ZooKeeper / 12.etcd 这一整支的祖宗;

同样 2006 年, Amazon 把 AWS 端上来 (S3 三月、EC2 八月): Amazon 自己零售业务的基建本来就得扛"黑五"那种弹性洪峰, 闲着也是闲着, 干脆把这份弹性算力租出去; 云 (IaaS) 就这么诞生了, 基础设施第一次变成"按需开、按量付费";

二、2006-2010: 开源复刻 + NoSQL 把数据库哲学掰弯

Google 论文写得漂亮但代码不开源, 于是全世界开始照着论文造开源版:

  • Hadoop (Yahoo, 2006, Doug Cutting): 就是开源版 GFS + MapReduce; 需求是 Yahoo 自己也要做 web 尺度搜索, 顺手把它开源, 结果成了整个大数据时代的地基;

与此同时 Amazon 摊上另一个需求, 把数据库哲学彻底掰弯了:

  • Dynamo (SOSP 2007): 需求是购物车的写入永远不能被拒绝, 哪怕机房之间网络断了也得能下单; 这逼着它放弃强一致, 选了"永远可用 + 最终一致" (AP); 它把 CAP 那个"一致性 vs 可用性"的取舍第一次落成了真系统, NoSQL 的世界观就此成形; 这条线接你的 6.cap 和 7.eventual_consistency;

Dynamo 一出, 一票公司照着自己的痛点造数据库:

  • Cassandra (Facebook, 2008): Dynamo 的去中心化 + Bigtable 的数据模型, 给 Facebook 收件箱搜索用;
  • ZooKeeper (Yahoo, 2008): 开源版 Chubby, 几乎冲着那篇论文做的; 后来 Kafka/HBase/Hadoop 全靠它协调;
  • Redis (2009) / MongoDB (2009): web 2.0 的应用要比 SQL 更快、更灵活的存储;

这一代的主题词就俩字: 分片 (sharding) 和最终一致, 把单机数据库扛不住的量摊到一堆机器上; 接 8.sharding;

三、2010-2012: 从批处理到实时, 顺便把强一致捡回来

MapReduce 是批处理, 跑一轮要很久; 移动和社交一爆发, 大家要的是实时 + 迭代:

  • Kafka (LinkedIn, 2010): 需求是 LinkedIn 内部一堆系统两两对接成了意大利面; 它干脆把"所有数据流"统一成一条持久化的日志, 谁要谁来订阅; 这是"log 是一等公民"这个思想的引爆点;
  • Spark (Berkeley AMPLab, 2010): 需求是机器学习要反复迭代, 而 MapReduce 每轮都落盘太慢; Spark 把中间结果搬进内存, 快了一个量级;
  • Storm (Twitter, 2011): Twitter 要实时处理推文洪流 (firehose);

然后 Google 又来"既要又要"了:

  • Spanner (OSDI 2012): 需求是 AdWords 这种业务既要全球规模、又要强一致、还要 SQL; 它用 TrueTime (GPS + 原子钟把时钟不确定性框死) 做到了全球外部一致; NewSQL 诞生, 它证明了"一致性和扩展性不是只能二选一"; 接你的 9.spanner;

四、2013-2016: 容器革命 + 编排之战

前面解决的都是"数据怎么存怎么算"; 这一代换了战场: 应用本身怎么打包、怎么部署、怎么管;

  • Docker (dotCloud, 2013): Linux 的 cgroups + namespace 其实早就有 (你 Docker 笔记里那套), 但难用; Docker 的贡献是给它配了镜像格式 + 仓库, 一举解决"在我机器上能跑、到线上就崩"这个世纪难题; 这是整个云原生的大爆炸点;
  • Kubernetes (Google, 2014): 需求很自然, 大家都有容器了, 可几千个容器谁来编排、谁来重启、谁来扩缩? Google 把内部 Borg 十五年的经验蒸馏成 k8s 开源出来; 它的声明式 + reconcile 范式接你的 12.etcd;
  • Raft (Stanford, ATC 2014): 需求是 Paxos 太难实现也太难讲 (Chubby 自己都写《Paxos Made Live》哭诉过); etcd/Consul 这些新协调系统需要一个能看懂、能写对的共识; 接 5.paxos;
  • AWS Lambda (2014): Serverless/FaaS, 需求是连服务器都不想管, 按调用次数付费;
  • etcd (CoreOS, 2013 起) + CNCF (2015): etcd 是 Raft 做的协调存储, 成了 k8s 的大脑; CNCF 成立、k8s 当种子项目, "云原生"这个旗号正式立起来;

这几年还有场著名的编排之战: k8s vs Docker Swarm vs Mesos; 到 2017 年 k8s 基本通吃;

五、2016-2020: 服务网格 + 云原生成熟

业务拆成几百个微服务之后, 新的痛点冒出来: 服务之间怎么通信、怎么加密、怎么观测; 每个服务自己写一套重试/TLS/埋点, 还是多语言各写一遍, 受不了;

  • Envoy (Lyft, 2016): 需求是 Lyft 一堆 PHP/Python/Go 混编的微服务, 要统一的网络治理和可观测性; 做法是把这些横切关注点抽进一个进程外代理 (sidecar); 它是 service mesh 的数据面;
  • gRPC (Google, 2016): HTTP/2 + protobuf 的 RPC, 给微服务做高效通信;
  • Istio (Google/IBM, 2017): 需求是管一大群 Envoy, 用声明式 CRD 统一下发流量/安全/观测配置; 它是 mesh 的控制面, 本质就是跑在 k8s 上的一组控制器;
  • Prometheus (CNCF, 2016 入会) / Jaeger (Uber): 微服务多到没法 debug, 于是指标和分布式追踪成了刚需; 这正是 Envoy 当年最初的动机 (观测优先);
  • eBPF / Cilium (2016+): 在内核里做网络/安全, 催生了 sidecarless 的苗头, 想甩掉每个 Pod 一个代理的开销;

到 2020 年, k8s 成了事实上的"云操作系统", 整套栈 (容器 → 编排 → 协调 → 微服务 → 网格 → 可观测) 基本定型, 云原生从前沿变成主流;

一条贯穿二十年的暗线

把这二十年抽象一下, 其实就是一个"撑爆 → 抽象 → 再撑爆"的循环:

被撑爆的东西 逼出的抽象 代表作
单机存不下 分布式文件/存储 GFS、Dynamo、Bigtable
单机算不动 分布式计算框架 MapReduce、Spark
关系库扛不住量 NoSQL / 分片 / NewSQL Cassandra、Spanner
协调/选主乱套 协调服务 Chubby、ZooKeeper、etcd
部署环境不一致 容器 Docker
容器太多管不过来 编排 Kubernetes
微服务通信乱 服务网格 Envoy、Istio
系统大了看不见 可观测性 Prometheus、Jaeger

还有一条更深的暗线值得记: 越往后, 工程界越把"故障是常态"刻进骨子里; 从 GFS 假设磁盘会坏, 到 Dynamo 宁可最终一致也要永远可用, 到 Netflix 用 Chaos Monkey 主动搞破坏, 到 k8s 默认 Pod 随时会死然后自愈; 这条"design for failure"的暗线, 一路通到今天的可靠性研究 (metastable failure、过载控制), 也就是这堆系统在压力下怎么别雪崩; 那才是这二十年留下的、最硬也最值得做的题;
